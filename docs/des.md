# Cup 2.0 设计文档-初稿

Cup 2.0 计划将对其可扩展性进行革新，由于C++本身的生态较为庞杂，因此Cup 2.0 将使用插件化的架构，以便于扩展功能。

# 模块划分

## 日志
日志系统提供4种级别的日志输出：
+ 调试日志(`LOG_DEBUG`): 输出调试信息，用于开发者调试，以发行模式编译时不会输出
+ 信息日志(`LOG_INFO`): 输出一般信息，用于程序运行过程中的信息
+ 警告日志(`LOG_WARN`): 输出警告信息，用于程序运行中出现的非致命错误
+ 错误日志(`LOG_ERROR`): 输出错误信息，用于程序运行中出现的致命错误,
    此类错误应以`std::runtime_error`抛出，并在`main`函数中统一处理

## TOML反序列化
对于TOML文件的反序列化应通过概念`Deserializable`和结构体`Deserializer`进行扩展，通过继承复用数据结构体

对于TOML的处理过程给出明确的字段缺失错误和字段类型不匹配警告

## CMake生成
CMake代码的生成使用专门的API(`cmake::Generator`)

## DLL插件接口
插件应继承接口`PluginInterface`并实现方法`execute()`

## 命令行参数解析
通过 `SysArgs` 进行参数解析，解析规则如下：
+ 对于参数之前没有任何`--`开头的参数，将其识别为普通位置参数
+ 对于参数前有`-`开头的参数，将其识别为标志选项，无论其在命令的除首位意外的任何位置都会被解析
+ 对于参数前有`--`开头的参数，将其识别为带参数选项，其后紧跟的参数将被视为该选项的参数，且保留位置顺序
+ 占位符选项`_`，该选项将被忽略，不会记录在`SysArgs`中，该选项使得带参数选项接受以`-`或`--`开头的参数成为可能

## 命令行程序调用

通过`Command`类进行命令行程序调用，该类允许对输出进行重定向，以读取程序的输出

## 资源访问器

通过`Resource`类进行资源访问，该类目前设计用于读写程序缓存目录(`$HOME/.cup`)
